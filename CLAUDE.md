# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

The `.csproj` sets `Platforms=x64;ARM64` and enables MSIX tooling, so **`Configuration`/`Platform` must always be specified together** — building with the bare "Any CPU" solution platform (or omitting `-p:Platform`) fails with either `MSB4126` or an MSIX packaging error (`NETSDK1XXX ... AnyCPU`).

```powershell
# Build the app only
dotnet build FastExplorer.csproj -c Debug -p:Platform=x64

# Build app + test project together (the .sln already maps its "Any CPU" solution
# platform to x64 per-project, so no -p:Platform flag is needed here)
dotnet build FastExplorer.sln -c Debug

# Run all tests
dotnet test FastExplorer.Tests\FastExplorer.Tests.csproj -p:Platform=x64

# Run a single test
dotnet test FastExplorer.Tests\FastExplorer.Tests.csproj -p:Platform=x64 --filter "FullyQualifiedName~ClassName.MethodName"

# Run the built app directly (dotnet run resolves the wrong output path for this
# project's platform-qualified bin folder — launch the exe instead)
.\bin\x64\Debug\net8.0-windows10.0.26100.0\FastExplorer.exe
```

If `dotnet` isn't resolving on PATH in a given shell, invoke it via its full path (`C:\Program Files\dotnet\dotnet.exe`) instead of troubleshooting PATH.

`FastExplorer.csproj` explicitly excludes `FastExplorer.Tests\` from its own `Compile`/`None` globs (SDK-style projects otherwise auto-glob the sibling test folder's `.cs` files into the app). If a new top-level folder is ever added alongside the test project, apply the same exclusion pattern.

## Distribution

**End users install via `install.ps1` (`irm .../install.ps1 | iex`), which downloads a self-contained release, not the framework-dependent build `dotnet build`/`dotnet publish` produce by default.** `FastExplorer.csproj`'s own defaults (`SelfContained=false`, `WindowsAppSDKSelfContained=false`, `WindowsPackageType=None`) are deliberately left alone for everyday `dotnet build`/local development — self-contained publishing is only ever invoked as command-line MSBuild property overrides (`-r win-x64 --self-contained true -p:WindowsAppSDKSelfContained=true`), in `.github/workflows/release.yml`, never baked into the `.csproj` itself. The distinction matters: a plain build/publish requires the .NET 8 Desktop Runtime *and* the Windows App SDK Runtime already present on the machine running the exe (fine for a dev's own machine, the original source of "muita coisinha para baixar" friction when a non-technical user tried it); the self-contained publish bundles both, at the cost of a much larger output (~210 MB vs. a few MB) — an explicit, deliberate trade-off in favor of "friend runs one command, nothing else to install" over download size.

**`release.yml` triggers on pushing a tag matching `v*`** (e.g. `git tag v0.2.0 && git push origin v0.2.0`), not on every push to `main` — cutting a release is a distinct, deliberate action from ordinary merges, unlike `ci.yml` (build+test on every push/PR to `main`). It re-runs the test suite before publishing (redundant with `ci.yml` in the common case, but guards against tagging a commit `ci.yml` never actually validated) then zips the self-contained `win-x64` publish output and attaches it to a GitHub Release via `gh release create --generate-notes`, using the pushed tag as both the release tag and version label. Only `win-x64` is built for releases even though the `.csproj` also supports `ARM64` (`RuntimeIdentifiers`) — kept out of scope until there's an actual ARM64 user, since it would double the release job's build time and artifact count for a currently-hypothetical audience.

**MSIX packaging (`EnableMsixTooling=true` in the `.csproj`) was considered and deliberately not used for distribution** — it's the more "native" Windows install experience (Start Menu entry, uninstaller in Settings, etc.), but a non-Store MSIX needs a trusted signing certificate or the installing machine has to manually trust a self-signed one first, which is *more* friction for a non-technical friend than a plain zip, not less, absent an actual paid code-signing certificate. `WindowsPackageType=None` stays as the project default for this reason; revisit only if a real certificate becomes available.

**Programmatic taskbar pinning (`install.ps1`'s `Verbs() | Where-Object {...} | DoIt()` COM automation via `Shell.Application`) is best-effort, not guaranteed, and the script says so explicitly rather than claiming success.** There is no supported public API for an *unpackaged* app to pin itself to the taskbar — this is the same informally-documented shell-verb trick used elsewhere in this codebase for automation (`RecycleBinService`, `ShellContextMenuTests`), but unlike those, Microsoft has progressively locked down unsanctioned taskbar pinning since roughly Windows 11 22H2 specifically to stop malware from doing exactly this, so it may silently no-op on a given machine even when the verb is found and `DoIt()` doesn't throw — "the verb ran without error" and "the pin actually appeared" are not the same guarantee. The script matches verb names in both Portuguese (`"Fixar na barra de tarefas"`) and English (`"Pin to taskbar"`/`"Pin to Taskbar"`) since the verb's display text is localized by the OS, then always prints the manual right-click fallback instruction regardless of whether the automated attempt looked like it worked.

## Architecture

**MVVM via CommunityToolkit.Mvvm source generators** (`[ObservableProperty]`, `[RelayCommand]`). `TabViewModel` is the workhorse — one instance per tab, owning navigation history, search, sort state, and all file operations. `MainViewModel` owns the tab collection and the favorites/theme state shown in the sidebar.

**View-to-view communication goes through `WeakReferenceMessenger`, never direct references.** `MainWindow` used to expose a `public static MainWindow.Instance` that `FolderView` reached into directly; that was removed in favor of messages defined in `Messaging/Messages.cs` (`PreviewRequestedMessage`, `AddFavoriteRequestedMessage`, `ThemeChangedMessage`). `MainWindow` registers handlers in its constructor and unregisters on `Closed`. Any new cross-view interaction should follow this pattern rather than reintroducing a singleton.

**Paste and drag-drop copy share one pipeline in `TabViewModel`:** `TransferAsync` runs on a background thread via `Task.Run`, accepts a `CancellationToken` (wired to the status bar's Cancel button, shared with `_operationCts`), and takes a `ConflictResolution` (`FailOnConflict` / `Replace` / `Skip` / `KeepBoth`). Conflict detection (`PeekPasteConflicts`/`PeekDropConflicts`) happens synchronously before the transfer starts; `FolderView` code-behind is what actually prompts the user via `ResolveConflictsAsync` (a `ContentDialog`) — the ViewModel never shows UI itself. `PasteAsync` and `CopyIntoAsync` (drag-drop) both route through this same helper; add new bulk copy/move operations here rather than duplicating the cancellation logic. `DeleteSelectedAsync` doesn't go through `TransferAsync` (deletion has no conflict/destination concept to share) but mirrors its shape — its own `Task.Run` loop, same `_operationCts`/`IsBusy` wiring — so the two should be kept in sync deliberately, not by shared code, when either changes (e.g. both now report `IsTransferring`/`TransferProgress` per-item for the status bar's determinate `ProgressBar`, added independently in each).

**"This PC" (drives list) is probed in parallel, not sequentially.** `FileSystemService.GetDrivesAsync` fires one `Task.Run` per `DriveInfo` with an independent 2-second timeout each (`DriveProbeTimeout`), streaming each resolved drive back via an `Action<FileSystemItem>` callback as soon as it's ready — a single unreachable network share or empty optical drive no longer stalls (or serially delays) the rest of the list. `TabViewModel.LoadDrivesAsync` is the caller; it guards against a stale drive arriving after the user has already navigated into a folder by checking `CurrentPath` is still empty before appending.

**`TabViewModel.IsAtDrivesRoot` (`CurrentPath` is empty) drives contextual UI in `FolderView.xaml`** — the address bar, search bar, and "New folder" button are only visible when actually inside a folder, via `Converters/BoolToVisibilityConverter` with `ConverterParameter=Invert`.

**`Services/*` are stateless static classes** (`FileSystemService`, `SearchService`, `FavoritesService`, `SettingsService`, `PropertiesService`, `NetworkDriveService`, `RecycleBinHelper`) and are the layer to unit test — `FastExplorer.Tests` covers them directly rather than the ViewModels/Views. `RecycleBinHelper` wraps `IFileOperation` via raw COM interop (not the legacy `SHFileOperationW`); its test (`FileSystemServiceTests.Delete_ToRecycleBin_ItemAppearsInRecycleBin`) independently verifies the deleted file actually lands in the real Recycle Bin using `Shell.Application` COM automation, separate from the interop under test — don't remove that verification if touching `RecycleBinHelper`, since a COM vtable/HRESULT mistake there fails silently otherwise.

**Cloud-sync status (OneDrive "Files On-Demand") is read from raw `FileAttributes` bits**, not any cloud API: `FileSystemService.ToItem` checks `FILE_ATTRIBUTE_RECALL_ON_DATA_ACCESS` (`0x400000`, placeholder/not downloaded) and `FILE_ATTRIBUTE_PINNED` (`0x80000`, always-available) directly on the raw attribute int, since `System.IO.FileAttributes` doesn't expose named members for them. `FileSystemItem.CloudGlyph`/`HasCloudStatus` drive the small overlay icon in `FolderView`'s item template.

**Segoe Fluent Icons glyphs are always built from raw codepoints** (`((char)0xE753).ToString()`), never typed as literal PUA characters in source — see the comments in `Models/IconGlyphMap.cs` and `Models/FileSystemItem.cs`. Editing a glyph as a literal character in a source file risks silent encoding corruption; add new glyphs the same way.

**Theme (`AppTheme`: System/Light/Dark)** is persisted via `SettingsService` (JSON in `%LocalAppData%\FastExplorer\settings.json`, same convention as `FavoritesService`) and applied by setting `RootGrid.RequestedTheme` in `MainWindow`. `RootGrid` has an explicit `Background="{ThemeResource ApplicationPageBackgroundThemeBrush}"` — without it the window's canvas stays black regardless of theme (only controls with their own opaque fill render; plain `TextBlock`s become invisible dark-on-black in Light mode). Don't remove that background.

**The live window icon and the `.exe`'s own icon are set independently.** `ApplicationIcon` in `FastExplorer.csproj` only covers the compiled `.exe` resource (File Explorer, shortcuts); the actually-running window's taskbar icon requires `AppWindow.SetIcon(...)` called explicitly in `MainWindow`'s constructor (`ApplyIcon()`), since this is an unpackaged WinUI 3 app with no packaging manifest to source it from otherwise.

**Properties dialog (`Controls/PropertiesView`) permissions are read-only by design** — it displays owner + ACL entries via `PropertiesService.GetAccessInfo` but never modifies them. The only mutation it performs is toggling Read-only/Hidden `FileAttributes` (`PropertiesService.SetAttributes`), applied from `FolderView.ShowPropertiesAsync` only when the dialog's primary button is pressed and the item isn't a drive. `GetAccessInfo`'s `AccessType`/`Rights` strings are translated to PT-BR by `PropertiesService.TranslateAccessType`/`TranslateRights` before being wrapped in `AccessRuleInfo` — `FileSystemRights` is a `[Flags]` enum, so `.ToString()` on it returns a comma-separated list of whichever *named* combination it resolves to (e.g. `"Modify, Synchronize"`), not a fixed set of values; `TranslateRights` splits on `", "` and maps each token independently through `RightsTokens` so an unmapped/future token degrades to its English name rather than vanishing.

**The Recycle Bin has a browser (`Controls/RecycleBinView`, opened from the sidebar's "Lixeira" button in `MainWindow.xaml.cs`'s `RecycleBinButton_Click`), separate from `TabViewModel`'s normal folder browsing.** The bin is a virtual shell namespace, not a real directory — `TabViewModel` assumes `CurrentPath` is a real folder (history, `FileSystemWatcher`, etc.), so rather than special-casing that, the browser is a self-contained `UserControl` hosted directly in a `ContentDialog`, listing/acting on items via `RecycleBinService`. `RecycleBinService.ListItems()` reuses the same late-bound `Shell.Application` COM automation as `TryRestore` (originally added only for Ctrl+Z undo) rather than the strictly-typed `IShellFolder`/`IContextMenu` interop in `ShellContextMenu.cs`, for the same reason as before: a wrong property/verb name here just fails silently, it can't corrupt memory. Every `RecycleBinService` call re-enumerates the bin from scratch instead of caching COM item handles across calls — `RecycleBinItemInfo` (in `Models/`) is a plain snapshot record, and `TryRestore`/`TryDeletePermanently` re-match a target item by recombining `DeletedFrom`+`Name` into `OriginalPath` and re-walking the bin, exactly like the original `TryRestore` did — simpler COM lifetime management than holding references open across a dialog session, and the bin is never large enough for the extra walk to matter. `EmptyRecycleBin()` is the one exception: it calls `SHEmptyRecycleBinW` directly (the same native API Explorer's own "Empty Recycle Bin" command uses) instead of iterating `ListItems()` + `TryDeletePermanently` per item, since that would be one native call replaced by N COM round-trips for no benefit. Permanent delete and empty are both irreversible, so `RecycleBinView` confirms each first — restore has no such confirmation, mirroring how `DeleteSelectedAsync` sends to the bin without confirming either, since it's the reversible direction. **That confirmation is an inline panel (`ConfirmPanel`/`ConfirmText`/`ConfirmYesButton`/`ConfirmNoButton` in the XAML, driven by a stashed `_pendingConfirmAction` delegate), not a second `ContentDialog`, and this is load-bearing, not style** — `RecycleBinView` is itself hosted *inside* `MainWindow`'s own `ContentDialog` (`RecycleBinButton_Click`), and WinUI only allows one `ContentDialog` open per `XamlRoot` at a time; the first version of this feature called `new ContentDialog{...}.ShowAsync()` for the confirmation exactly like `FolderView`'s rename/conflict prompts do (which is safe there, since `FolderView` is never itself inside a dialog), and that threw on the second, nested `ShowAsync()`. Because the click handlers were `async void` and the app has no `Application.UnhandledException` handler, that exception didn't just fail the confirmation — it crashed the entire process, surfacing confusingly on the *next* UI interaction rather than immediately. Any future dialog-triggered-from-a-dialog UI in this app needs to use this same inline-panel approach (or close the hosting dialog first), never a second `ContentDialog` on the same `XamlRoot`.

**Disk usage analysis (`Controls/DiskUsageView`, a WinDirStat-style treemap — nav-bar button and per-folder context menu item, both "Analisar espaço em disco") scans one folder level at a time, not the whole subtree up front.** `Services/DiskUsageService.ScanLevelAsync` computes a total size for every *direct* child of the folder being shown (recursive for subdirectories, via `PropertiesService.CalculateFolderSizeAsync` — already-tested, already-cancellable, reused rather than duplicated), and drilling into a directory cell just re-scans at that new level. This bounds any single scan to "one folder's immediate children," never "however much of the disk is under here" — important since the obvious use case (right-click a suspiciously large folder like `node_modules`) is exactly the kind of folder an eager full-subtree walk would be slowest on. Subdirectory sizes are computed in parallel (`Task.WhenAll`), mirroring the same "fan out independent I/O" approach `FileSystemService.GetDrivesAsync` already uses for probing drives, and even the initial `Directory.EnumerateDirectories`/`EnumerateFiles` listing runs inside `Task.Run` rather than synchronously on the calling (UI) thread — unlike `TabViewModel.LoadFolder`'s ordinary enumeration, which doesn't bother, because a folder someone reaches for *this* feature to investigate is disproportionately likely to have an enormous flat file count.

**The treemap layout (`Models/TreemapLayout.cs`) is a from-scratch implementation of the squarified treemap algorithm (Bruls/Huizing/van Wijk), not a naive proportional slice-and-dice.** A simple "size ∝ rectangle width, stacked in one direction" layout produces unreadable slivers the moment the size range is wide (one large file next to a pile of tiny ones — which is the normal shape of a real folder), since a sliver a few pixels wide is both hard to see and hard to click. The squarified variant instead greedily packs items into rows/columns chosen to keep each cell's aspect ratio as close to square as possible. It's covered by `FastExplorer.Tests/TreemapLayoutTests.cs` (area conservation, bounds containment, aspect-ratio sanity under a large size spread) since it's the one piece of this feature that's pure, non-UI algorithmic logic worth verifying in isolation — everything else in `DiskUsageView` is UI/IO glue in the usual untested-Views style this repo already follows. `DiskUsageView.CanvasWidth`/`CanvasHeight` constants **must** match `TreemapCanvas`'s declared `Width`/`Height` in the XAML exactly, since the algorithm needs real pixel bounds up front — reading `ActualWidth`/`ActualHeight` instead would race the control's first layout pass (return 0 before it's measured).

**Treemap cells are colored by reusing `IconGlyphMap.GetColorHex`, via a throwaway `FileSystemItem` built just for that lookup** (`DiskUsageView.ToPseudoItem`) — `DiskUsageEntry` (the treemap's own lighter record: name/path/size/`IsDirectory`, no `FileSystemItem`'s thumbnail/cloud-status/etc. baggage) isn't a `FileSystemItem` and was deliberately kept that way, but constructing one on the fly to borrow the *existing* per-extension color table means the treemap's colors match the rest of the app's icons for free, with zero new color-mapping code to maintain. Deleting a selected cell reuses `RecycleBinHelper.SendToRecycleBin` directly (an `internal` method, reachable since `DiskUsageView` is in the same assembly) rather than going through `FileSystemService.Delete(FileSystemItem, ...)`, for the same "avoid a throwaway `FileSystemItem` just to call an API shaped around one" reason — and, like `RecycleBinView`'s permanent-delete/empty actions, its confirmation is `DiskUsageView`'s own inline `ConfirmPanel`, not a nested `ContentDialog`, since this view is hosted inside one already (see the Recycle Bin entry above for why that specific mistake crashes the whole app, not just the dialog). Deleting from here does **not** push an `UndoService` action — same simplification `RecycleBinView` already makes for its own deletes, not a new gap introduced here.

**`Services/ShellContextMenu.cs` is the riskiest interop in this codebase** — it queries the real Windows Shell's `IContextMenu` (via `IShellFolder.GetUIObjectOf`) so third-party shell extensions (7-Zip, antivirus scanners, "Open in ..." tools) appear in our own context menu instead of us reimplementing each one. It's deliberately scoped down from what a full native integration would do:
- Single-item selection only (`FolderView.AddShellExtensionItems` bails if more than one item is selected) — no PIDL-array marshaling for multi-select.
- Text-only recreation — menu items are read via `GetMenuItemInfoW`/walked recursively for submenus and rebuilt as our own `MenuFlyoutItem`/`MenuFlyoutSubItem` tree (inserted into `FolderView.xaml`'s `MenuFlyout` at the `ShellExtensionsAnchor` separator, cleaned up on `Closed`), rather than displaying the shell's native `HMENU` directly via `TrackPopupMenuEx` — no owner-drawn icons/bitmaps, no `IContextMenu2`/`IContextMenu3` message forwarding.
- Standard verbs we already implement ourselves (open/cut/copy/paste/delete/rename/properties) are filtered out via `GetCommandString(GCS_VERBW)` so the shell's version never shadows our own async/cancellable/conflict-resolved file-op pipeline — anything without a resolvable verb string (i.e. almost all third-party extension entries) passes through.
- Every COM/P-Invoke call is wrapped defensively (`ShellContextMenu.TryCreate` returns `null` on any `COMException`/etc.) so a misbehaving extension degrades to "no extra items," never a crash. `ShellContextMenu.Dispose()` is idempotent (guarded by `_disposed`) since both `Opening` and `Closed` handlers can trigger cleanup.
- The `IShellFolder`/`IContextMenu` interface declarations must match the native vtable order *exactly*, including methods this code never calls (e.g. `GetDisplayNameOf`) — those are declared with dummy-but-syntactically-valid signatures purely to occupy the correct vtable slot. Getting the order or a struct's field layout wrong causes memory corruption (`AccessViolationException`, uncatchable), not a clean .NET exception, so verify any change here against a real `shobjidl_core.h`/MSDN reference and re-run `ShellContextMenuTests` before trusting it. That test suite deliberately exercises this against real files/folders on the host machine.

**Tests that exercise real Shell COM automation against the host's actual desktop/Recycle Bin (`ShellContextMenuTests`, `RecycleBinServiceTests`, and `FileSystemServiceTests.Delete_ToRecycleBin_ItemAppearsInRecycleBin`) are tagged `[Trait("Category", "RequiresDesktop")]` and excluded from CI (`ci.yml`/`release.yml`'s `--filter "Category!=RequiresDesktop"`), run only locally.** These calls were observed to *hang indefinitely* (not fail) on a GitHub Actions Windows runner — one run sat stuck for nearly two hours before being manually cancelled, with `dotnet test` never producing a single result past "test files matched." The runner apparently lacks whatever a genuine interactive desktop session provides for these STA-affine shell COM objects (`Shell.Application`, `IShellFolder`/`IContextMenu`) to complete cross-apartment calls — consistent with these tests' own pre-existing doc comments, which already said they needed "a real machine," just not that a CI runner would *hang* rather than merely fail. Both workflows also pass `--blame-hang-timeout 120s --blame-hang-dump-type mini` as a safety net so any *other*, still-undiscovered hang fails fast with a diagnostic dump instead of silently consuming Actions minutes for hours. Any new test that touches real Shell/Recycle Bin COM automation needs the same trait, and CLAUDE.md's own documented local test commands (`dotnet test FastExplorer.Tests\...` with no filter) remain the authoritative "run everything" command — the CI filter is additive, not a replacement.

**Both workflows pin `runs-on: windows-2022`, not `windows-latest`.** After excluding the `RequiresDesktop` tests above, the *next* CI attempt still failed - but this time with "Test host process crashed" (a genuine crash, not a hang), and the blame data collector's list of in-flight tests at the moment of the crash included `TreemapLayoutTests.Compute_MultipleItems_TotalAreaMatchesBoundsArea` and `IconGlyphMapTests.GetColorHex_ForUnknownExtension_ReturnsNull` — both pure computation with zero OS/COM/file-I/O interaction of any kind. That ruled out this repo's own test code as the cause a second time and pointed at the runner image itself: `windows-latest` had resolved to `windows-2025-vs2026`, a very recently released image at the time. `windows-2022` is a long-established, far more battle-tested GitHub-hosted image; pinning to it avoids riding the newest image's inevitable early rough edges. If CI/release ever needs a newer image again (e.g. for a future .NET/Windows SDK requirement), treat that as a deliberate, tested upgrade, not a side effect of `windows-latest` silently rolling forward underneath this repo.

**Per-app settings (theme + show-hidden-files) live in one `settings.json`** via `SettingsService`'s private `SettingsData`/`LoadData`/`SaveData` read-modify-write helpers — adding a new persisted setting means adding a field there plus typed `LoadX`/`SaveX` wrappers, never a separate file, so existing settings aren't clobbered.

**Cross-tab settings changes broadcast via messages, not a shared static.** `MainViewModel.ShowHiddenFiles` sends `ShowHiddenFilesChangedMessage` on change; each `TabViewModel` registers for it in its constructor (and unregisters in `Dispose`) and calls `Refresh()` to re-enumerate with the new flag. `FileSystemService.EnumerateDirectory`'s `includeHidden` parameter controls this via `EnumerationOptions.AttributesToSkip`. This mirrors the existing `ThemeChangedMessage` pattern rather than introducing a new communication style.

**Accent color recolors WinUI's built-in `SystemAccentColor*` resources, not our own controls.** `AccentColorPalette` holds the 8 base colors and computes the Light1-3/Dark1-3 tints WinUI's default control templates expect (ListView selection highlight, CheckBox/ComboBox focus states, etc. all key off these). `MainWindow.ApplyAccentColor` sets all seven resources on `Application.Current.Resources` and then briefly flips `RootGrid.RequestedTheme` to its opposite and back — existing controls don't repaint just because a resource *value* changed, and this forces WinUI to re-evaluate `ThemeResource` lookups. Persisted/broadcast the same way as theme (`SettingsService.LoadAccentColor`/`SaveAccentColor`, `AccentColorChangedMessage`).

**`x:Bind` with an inline `Converter=` inside a `DataTemplate` fails to compile when the containing XAML file's code-behind is a `Window`** (not a `Page`/`UserControl`/other `FrameworkElement`) — the compiler generates `bindings.SetConverterLookupRoot(this)`, and `this` (the `Window`) doesn't satisfy the `FrameworkElement` parameter, so it's a hard compile error (`CS1503`), not a runtime issue. Hit this in `MainWindow.xaml`'s accent-color `ComboBox.ItemTemplate`; the fix was switching that one template from `x:Bind`/`x:DataType` to classic `{Binding}` (which has no such restriction). `FolderView.xaml` and other `UserControl`-rooted views don't have this problem and should keep using `x:Bind` as usual - this only bites templates declared directly inside `MainWindow.xaml`.

**Image thumbnails load lazily per realized row, never per folder.** `FileSystemItem` inherits `ObservableObject` specifically so its `Thumbnail` property (nullable `BitmapImage`, absent everywhere else in the model) can notify the UI once decoded. The actual load is triggered by `FolderView.FileList_ContainerContentChanging` — WinUI's own "container about to show a new/recycled item" virtualization callback — via `args.RegisterUpdateCallback`, and `ThumbnailService.LoadThumbnailAsync` decodes at `DecodePixelWidth = 32` so even a multi-megapixel photo is cheap. This only ever runs for currently-visible image rows; it must not be moved into `TabViewModel.LoadFolder` or any other eager, whole-folder path.

**`FolderView.FileListArea`'s forced-opaque background (see the empty-folder-click saga below for *why* it must be opaque at all) is kept in sync with the theme via `ThemeChangedMessage`, not `ActualThemeChanged` or any resource lookup — and getting this wrong took five attempts, documented in full in `FolderView.ApplyOpaqueFileListAreaBackground`'s comment.** The short version: `Application.Current.Resources["ApplicationPageBackgroundThemeBrush"]` isn't scoped to this element's actual per-subtree theme (set via `MainWindow`'s `RootGrid.RequestedTheme`, a per-subtree override, not an app-global setting) and never updates after the first theme it happens to resolve to; indexing `Application.Current.Resources.ThemeDictionaries[...]` directly rests on a wrong assumption about where that resource actually lives; and even `this.ActualTheme`/`ActualThemeChanged` — which should be the most authoritative, zero-lookup signal available — was observed stale by exactly one toggle in testing, apparently because `FolderView` is reached through dynamically-composed content (`TabView`'s `TabItemTemplate`, itself hosted by `PaneGroupView`'s `TabView`, reached via `PaneNodeTemplateSelector`'s `ContentControl`) rather than a plain static XAML tree, and WinUI's theme cascade doesn't reliably propagate `ActualThemeChanged` through that composition the way a declarative `{ThemeResource}` binding does (which is exactly why `MainWindow`'s own `RootGrid` background, a plain XAML binding rather than a C# event handler, has always worked correctly). The fix: `ApplyOpaqueFileListAreaBackground` takes an `AppTheme` parameter carried directly as *data* in `ThemeChangedMessage`'s payload (or read once from `SettingsService.LoadTheme()` at construction) and matches it to a hardcoded color pulled from WinUI's own `generic.xaml` (`#202020` Dark / `#F3F3F3` Light) — no live property read of this element's own visual-tree state at all, for the two cases (`AppTheme.Light`/`Dark`) that actually matter. `AppTheme.System` is the one remaining exception that still asks `ActualTheme`, since "whatever Windows is currently set to" has no fixed answer otherwise. If a similar per-tab/per-pane visual needs to react to theme changes in the future, reach for `ThemeChangedMessage` first, not `ActualThemeChanged` — this composition doesn't carry the latter reliably.

**List vs. large-icons view is a runtime template/panel swap on a single `ListView`, not two separate controls.** `FolderView.xaml` defines `ListItemTemplate`/`LargeIconItemTemplate` and `ListItemsPanel`/`GridItemsPanel` as named `UserControl.Resources`; `ApplyViewMode` swaps `FileList.ItemTemplate`/`ItemsPanel` and toggles the `ListHeader` (sort-header row) visibility. `TabViewModel.ViewMode` is per-tab and in-memory only (no persistence, resets to `List` on a new tab) — if that's ever meant to persist, thread it through `SettingsService` like `ShowHiddenFiles`.

**Sorting has two independent entry points into the same `TabViewModel` state (`SortField`/`SortDescending`), and they deliberately have different click semantics.** `ListHeader`'s column buttons (Nome/Tamanho/Modificado, `FolderView.SortHeader_Click` → `TabViewModel.SortByCommand`) toggle direction when the *already-active* column is clicked again, matching Explorer's header-click convention. The nav bar's "Ordenar" flyout (`SortButton`, `RadioMenuFlyoutItem`s for field and direction, `FolderView.SortFieldItem_Click`/`SortDirectionItem_Click` → `SetSortFieldCommand`/`SetSortDescendingCommand`) sets each independently instead, since re-selecting "Nome" from a menu shouldn't flip direction the way re-clicking a column header does — these are two separate `[RelayCommand]`s on `TabViewModel`, not one shared toggle path. The flyout also exposes `SortField.Created` (date created), which has no corresponding column — adding a 5th list column for something Explorer itself only exposes via a menu, not a column, wasn't worth the layout churn. Because the flyout's `RadioMenuFlyoutItem`s only update their own `IsChecked` when *they* are clicked (not when sorting changes via the column headers), `SortFlyout_Opening` re-syncs all of them to the ViewModel's actual current state every time the flyout opens — without it, sorting via a column header would leave the flyout showing a stale, incorrect selection next time it's opened.

**Filtering by item type (`TabViewModel.FilterCategory`, `FileTypeCategory` enum, "Filtrar" flyout next to "Ordenar") reuses `IconGlyphMap`'s existing extension→icon table rather than a second extension list.** `IconGlyphMap.IconStyle` gained a `Category` field alongside `Code`/`ColorHex`, and `IconGlyphMap.GetCategory(FileSystemItem)` (exposed as `FileSystemItem.Category`) is the single source of truth both the icon color and the filter now read — adding a new extension only ever needs one dictionary entry updated, not two kept in sync by hand. `TabViewModel.ApplyFilter` runs before `OrderItems` in `LoadFolder` (so `ReconcileItems`' path-set diff naturally adds/removes rows when the category changes, same as any other data change) and is also applied inline in `RunSearchAsync`'s `OnResult` callback, so switching categories filters live search results the same way it filters an ordinary folder listing. `FileTypeCategory.All` is a passthrough (no `.Where()` allocated) rather than an always-true predicate, since it's the default/common case.

**Undo is a single app-wide stack (`Services/UndoService.cs`), not per-tab**, mirroring how Explorer's undo history is shared across the whole process. Every mutating `TabViewModel` operation (`NewFolder`, `NewFile`, `RenameAsync`, `DeleteSelectedAsync`, and `TransferAsync`'s callers `PasteAsync`/`CopyIntoAsync`) pushes a matching `UndoAction` (`CreateItemUndoAction`, `RenameUndoAction`, `DeleteUndoAction`, `MoveUndoAction`, `CopyUndoAction`) after it succeeds, and `TabViewModel.UndoCommand` (Ctrl+Z) pops and reverses the most recent one on a background thread. `DeleteUndoAction.Undo()` restores from the Recycle Bin via `RecycleBinService.TryRestore`, which uses **late-bound** `Shell.Application` COM automation (`Type.GetTypeFromProgID` + `InvokeMember`, matching `ShellContextMenuTests`'s verification style) rather than the strictly-typed vtable interop in `ShellContextMenu.cs` — a wrong property/verb name here just fails to find/restore the item, it can't corrupt memory the way a vtable mistake could. `RecycleBinServiceTests.TryRestore_AfterDelete_BringsFileBack` is the regression test for this; it was empirically verified to work before being wired into the undo stack, not assumed.

**Cut/copy/paste feedback is `StatusText` plus a per-item `IsCut` dim, not a toast/dialog.** `TabViewModel.CopySelected`/`CutSelected` set `StatusText` to a "Copied/Cut 'name'" summary and broadcast `ClipboardChangedMessage` (`Messaging/Messages.cs`) so every open tab — not just the one that triggered it — re-evaluates `FileSystemItem.IsCut` for its currently loaded `Items` via `TabViewModel.UpdateCutMarkers()`. That same method also runs at the end of every `LoadFolder` (covering `Refresh()`, navigation, and paste), so newly-loaded folders immediately reflect the current clipboard. `IsCut` is bound in `FolderView.xaml`'s `ListItemTemplate`/`LargeIconItemTemplate` via `Converters/BoolToOpacityConverter` (0.5 when cut) — mirroring Explorer's dimmed "about to move" look. `PasteAsync`/`CopyIntoAsync` set a "Pasted/Copied N items" `StatusText` on success (previously only failures produced any status message, which read as if nothing had happened).

**Cut/copy/paste/delete/rename/undo/select-all are keyboard shortcuts on the `FileList` `ListView` itself (`FolderView.FileList_KeyDown`), not window-level `KeyboardAccelerator`s** — they only fire while the list has keyboard focus, which WinUI does not grant automatically just because a folder loaded. `FileList_Loaded` calls `FileList.Focus(FocusState.Programmatic)` once so the shortcuts work immediately after opening/switching a tab without requiring a prior click. The context menu's `KeyboardAcceleratorTextOverride` (Ctrl+X/C/V, F2, Del, Alt+Enter) only renders the hint text next to each `MenuFlyoutItem` — it's display-only and intentionally not a real `KeyboardAccelerator`, so it can't double-fire alongside `FileList_KeyDown` when the context menu happens to be open.

**Type-ahead ("press a letter, jump to the matching item") is wired to `FileList`'s `CharacterReceived` event (`FolderView.FileList_CharacterReceived`), not folded into `FileList_KeyDown`'s `VirtualKey` switch above it.** `CharacterReceived` hands over the actual typed Unicode character rather than a `VirtualKey`, which matters here specifically because it's the only one of the two that correctly handles accented PT-BR file names (ç, ã, é...) and Shift/layout differences without hand-mapping `VirtualKey.A`..`Z`/digits; it also simply doesn't fire for Ctrl-chord shortcuts (Ctrl+C, Ctrl+V, etc.), so there's no risk of it double-handling anything `FileList_KeyDown` already owns. Runs entirely off already-loaded `vm.Items` in memory — no disk access, no debounce/timer beyond the idle-reset check — so it adds no ongoing cost to browsing.

**The type-ahead state machine only enters "cycling" mode after the *second* consecutive press of the same letter, and a `confirmedCycling` flag governs both matching and resetting — collapsing that distinction is what caused a real, shipped bug.** The first version tracked only "does the accumulated prefix consist entirely of one repeated character," which is trivially true for a single keystroke too — so after cycling on "d" a few times (prefix stayed `"d"`, never grew, by design) and then pressing a genuinely different letter like "p", the code appended instead of resetting (`_typeAheadPrefix.All(c => c == typed)` was false, so it fell into the "extend the prefix" branch), producing dead strings like `"dp"` that matched nothing — the search then did nothing at all until `TypeAheadResetInterval` (1s) elapsed and the *next* keystroke started fresh, which read to the user as "a delay when switching key." The fix (`confirmedCycling = withinBurst && _typeAheadPrefix.Length >= 2 && all-same-char`) only treats a burst as "confirmed cycling" from the second identical keystroke onward; a different letter arriving while `confirmedCycling` is true resets `_typeAheadPrefix` to just that new letter instead of appending. A *single* leading keystroke is deliberately left ambiguous (not yet "confirmed") so two different letters typed quickly (e.g. "d" then "o") still correctly extend into a multi-character search for "do" (Documentos) rather than being treated as a reset — only a *repeat* of the same letter first establishes cycling, and only breaking out of an *established* cycle resets. Any future change to this state machine should be checked against both of these cases together (repeat-then-different-letter, and two-different-letters-in-a-row), not just one — they pull in opposite directions and it's easy to fix one while re-breaking the other.

**Open tabs are restored on the next launch, saved once on shutdown, not on every navigation.** `TabViewModel`'s constructor takes an optional `initialPath`; when non-null and `Directory.Exists`, it calls the same `NavigateTo(path, recordHistory: false)` used for ordinary double-click navigation instead of `NavigateHome()`. `MainWindow`'s `Closed` handler calls `MainViewModel.SaveOpenTabs()`, which snapshots every tab's current `CurrentPath` (including the empty string for a tab sitting at "This PC") into `SettingsService.SaveOpenTabs`/`OpenTabs`, following the same `SettingsData` read-modify-write pattern as the other settings. `MainViewModel`'s constructor reads it back and opens one tab per saved path (falling back to a single fresh tab if the list is empty or every path is now gone) — this only touches the filesystem for a single `Directory.Exists` check and a normal folder enumeration per restored tab, so it adds no ongoing overhead, just a one-time read on startup and a one-time write on shutdown.

**Split-screen panes are a recursive tree, not a fixed layout** (`ViewModels/PaneNode.cs`): `MainViewModel.RootPane` is a `PaneNode` — either a leaf `PaneViewModel` (an independent tab strip, what `TabViewModel`s actually live in) or a `SplitPaneNode` (two `PaneNode` children + `Orientation`). Every node carries a `Parent` back-reference so an emptied leaf can be spliced out of the tree in O(1) (`MainViewModel.CollapseEmptyPane` promotes its sibling into the parent's old slot) without rebuilding the tree from the root, which would force WinUI to re-realize the whole visual subtree. `PaneViewModel.Owner` is a `required` back-reference to the single `MainViewModel`, set at construction — this is how `Controls/PaneGroupView` (bound to just a `PaneViewModel` via `DataContext`) reaches `NewTab`/`CloseTab`/`SplitPane`/`SetActivePane` without an ambient singleton; `ElementName` bindings back to `MainWindow`'s `RootGrid` (the pattern `FavoritesList`'s item template uses) don't resolve from inside a separate `UserControl`'s own XAML namescope, so this object-graph reference is the substitute. Split layout is session-only, not persisted — `SaveOpenTabs()` flattens every leaf pane's tabs into one list, same shape as before this feature existed, and `RootPane` always starts as a single unsplit pane on the next launch.

**Rendering the pane tree needs a `DataTemplateSelector`, not implicit XAML templates** (`Converters/PaneNodeTemplateSelector.cs`, registered globally in `App.xaml` since `Controls/SplitContainerView.xaml.cs` needs to resolve it too, for its own nested children) — WinUI, unlike WPF, has no keyless/`DataType`-matched implicit template lookup for `ContentControl.Content`, so a `SplitPaneNode` vs. leaf `PaneViewModel` is dispatched to `SplitContainerView`/`PaneGroupView` explicitly via the selector rather than "just working" through resource lookup.

**`SplitPaneNode.First`/`Second` are `[ObservableProperty]`, not plain properties with a hand-written setter — this is load-bearing, not style.** `SplitContainerView` sets each child `ContentControl.Content` once in `BuildLayout()` (an imperative C# assignment, not a live XAML binding), then subscribes to the bound `SplitPaneNode`'s `PropertyChanged` and updates just the affected `ContentControl.Content` when `First`/`Second` fires — that subscription is the *only* thing that repaints a pane after `SplitPaneNode.Replace()` (used whenever a split targets an already-nested pane, not the tree's root) mutates an existing, already-rendered node's child in place. Before `First`/`Second` were observable, that mutation was completely silent: nothing was listening, so splitting an already-split pane updated the tree data correctly but never repainted it — and it only *appeared* to work in cases where an unrelated side effect (e.g. the drag's source pane happening to empty out and collapse) reassigned `MainViewModel.RootPane` itself as a side effect, which does trigger a full rebuild via `MainWindow`'s `ContentControl` binding. That made the bug intermittent and confusing (splitting one side of an existing split would work, the other wouldn't, depending on incidental collapse behavior) rather than a clean, obviously-always-broken repro.

**`SplitContainerView` builds its `Grid.RowDefinitions`/`ColumnDefinitions` in code-behind, not XAML, and seeds them from the model once rather than binding.** Which dimension is used depends on `SplitPaneNode.Orientation`, only known once the node is bound (`DataContextChanged`) — and `RowDefinition`/`ColumnDefinition` in WinUI 3, unlike UWP, don't derive from `FrameworkElement`, so they have no `SetBinding`; a live two-way binding to `FirstLength`/`SecondLength` isn't available the way it would be in WPF. This isn't a problem in practice: `CommunityToolkit.WinUI.Controls.Sizers`' `GridSplitter` (added via `PackageReference` for this feature) resizes the Grid's own `RowDefinition`/`ColumnDefinition` objects directly during a drag, independent of any binding — the model's `FirstLength`/`SecondLength` are a one-time initial size only, since split ratios aren't persisted (see above).

**Drag-to-dock (drag a tab to an edge of another pane to split, or to the center to move it into that pane's strip) is built on `TabView`'s native tab-drag events, not a custom drag implementation** (`Controls/TabDragState.cs`, `Controls/DockZone.cs`, `PaneGroupView.xaml.cs`). `TabView.CanDragTabs="True"` + `TabDragStarting` starts a real OS-level `DataPackage` drag (the same mechanism used elsewhere for file drag-drop), which is why other `AllowDrop` elements outside that `TabView` — specifically, each `PaneGroupView`'s own `DragOverlay` — receive `DragOver`/`Drop` for it. The dragged `TabViewModel`/source `PaneViewModel` travel through the static in-proc `TabDragState` (simpler than round-tripping through the `DataPackage`, which just carries a boolean marker format so `DragOver` handlers — including `FolderView`'s own unrelated OS-file-drop handling — can cheaply tell this apart from an ordinary file drag). `DragOverlay` is `Visibility="Collapsed"` (and therefore non-hit-testable) except during an active tab drag — toggled via `TabDragState.DragStarted`/`DragEnded`, which every live `PaneGroupView` subscribes to — otherwise it would permanently sit on top of the file list and swallow every click. Drop zone (edge vs. center) is computed from pointer position as a fraction of the overlay's own bounds (`PaneGroupView.ComputeZone`, 25% edge bands). `TabView.TabStripDrop`'s delegate is a plain `DragEventHandler` (`object sender`), unlike `TabDragStarting`/`TabDroppedOutside` which are `TypedEventHandler<TabView, ...>` — mismatching this is a compile error (`CS0123`), not a runtime one. **`TabDragState.End()` must run on every code path that can end a drag, including a no-op one** — `PaneTabView_TabStripDrop` originally early-returned when a tab was dropped back onto its own source pane's strip (an ordinary same-pane reorder, handled natively by `CanReorderTabs`), which skipped `End()` entirely; since neither `TabStripDrop` nor `TabDroppedOutside` fires for that case, `DragOverlay` was left permanently `Visible` (and hit-testing on top of the `TabView`/`FolderView` beneath it, since it renders after them in z-order) after the very next ordinary tab reorder — silently swallowing all clicks, right-clicks, and drops on that pane with no visible sign anything was wrong, since the overlay itself is fully transparent. `RootGrid_PointerPressed` also clears `TabDragState` defensively if it's ever still non-null during a plain click (a real OS drag can't be in progress during ordinary click routing), as a safety net against any other leak of this kind. **`MainViewModel.SplitPaneWithTab` (not raw `RemoveTabFromPane` + `SplitPane` called separately) is what `DragOverlay_Drop`'s edge-zone branch must call — order matters.** Removing the dragged tab from its source pane first, the way the code originally worked, can collapse that source pane out of the tree *before* the split runs, if it was the source's last tab and source == target (splitting a pane against its own single tab): `SplitPane` would then reparent under a `Parent` reference that had just been spliced out and no longer led anywhere reachable from `RootPane`, and the pane would simply vanish. `SplitPaneWithTab` removes the tab from the plain list (no collapse check yet), splits while the target's tree position is still valid, and only *then* checks whether the source needs to collapse — safe for both the self-split and cross-pane-drag cases. A single-tab pane dropped on its own edge is treated as a no-op (nothing would be left on one side).

**The drop-zone feedback is a fixed, pane-centered "dock guide" — a plus-shaped cluster of 5 small chips (`DockGuide`/`DockGuideLeftChip`/`...RightChip`/`...TopChip`/`...BottomChip`/`...CenterChip` in `PaneGroupView.xaml`) — not a resized colored `Border`.** This replaced an earlier version that resized one `Border` to preview the actual resulting split size (half the pane) with a text label naming the action; that was replaced because it didn't match the Windows 11 File Explorer/Visual Studio docking-guide convention the app was asked to mirror. Critically, **only the feedback changed — `ComputeZone` (25% edge-band distance check against the raw pointer position) is byte-for-byte the same detection logic as before**, deliberately, to avoid touching the one part of this feature that's already been debugged through several rounds (see `SplitPaneWithTab` and `SplitPaneNode.First`/`Second` above). `PaneGroupView.UpdateDockGuide(DockZone)` just highlights whichever one of the 5 chips matches the zone `DragOverlay_DragOver`/`_Drop` already computed, via `SetChipActive` swapping each chip's `Background`/`BorderBrush` (and its child `FontIcon`'s `Foreground`, for the 4 directional chips — the center chip has no icon, just a plain filled square) to the accent brush when active, `ClearValue()`-ing them back to the shared `DockGuideChipStyle` when not. The chips sit at a **fixed** position (pane-centered, not following the cursor) — real docking-guide UIs work this way too; a cursor-following guide would be both harder to hit reliably while dragging and more code for no real benefit.

**`PaneGroupView`'s active-pane tracking must use `AddHandler(PointerPressedEvent, handler, handledEventsToo: true)`, not a plain XAML `PointerPressed=` attribute.** `ListView`/`TabView` mark pointer-pressed handled internally once they've processed selection, which stops the event from bubbling any further up the visual tree — a plain attribute-wired handler on the pane's root `Grid` would silently never fire for clicks on an actual file or tab (only truly empty space), leaving `MainViewModel.ActivePane` stuck and making sidebar actions (favorites, "This PC") appear to do nothing once more than one pane is open.

**Dragging a file out of `FileList` (e.g. onto another pane to copy it there) needs `DragItemsStarting`, and the `StorageItems` payload can't be built synchronously.** `FolderView.xaml`'s `FileList` sets `CanDragItems="True"`; `FileList_DragItemsStarting` can't `await` inline (the event args offer no such hook), so it resolves each dragged item's path to a real `IStorageFile`/`IStorageFolder` inside a `DataPackage.SetDataProvider` callback instead, deferred until an actual drop target asks for the data. This only ever copies (never moves), matching how drops from outside the app already behave.

**`TabView`'s own default style sets `VerticalAlignment="Top"` — it does not stretch to fill its container.** This is defined in `DefaultTabViewStyle` in the WinUI `generic.xaml` itself (not something visible from this codebase), so `Controls/PaneGroupView.xaml`'s `PaneTabView` now sets `VerticalAlignment="Stretch"` explicitly to override it. Without that override, `TabView` (and everything inside its selected tab's content, i.e. the whole `FolderView`) sizes itself to its *natural content height* and top-aligns within `PaneGroupView.RootGrid`, rather than filling the available pane height. This is invisible with a real folder — a `ListView`'s natural desired height scales with item count, so it's normally tall enough to fill the pane anyway — but glaring for a genuinely empty one: `TabView` shrinks to a thin sliver just tall enough for the tab strip plus the file list's header, and everything visually "below" that is actually *outside* `FolderView`'s bounds entirely — just `RootGrid`'s own same-colored background showing through behind it, not part of any interactive element. This silently ate every click, right-click, and file drop on an empty folder's blank area, while the exact same gestures worked fine on a real item, the header row, the nav bar, and the address bar (all still genuinely part of `FolderView`'s shrunk sliver). That pattern looked exactly like a `ListView`-specific gesture-routing gap, so several rounds of fixes targeted the wrong layer entirely before this was found by isolating each layer with a throwaway solid `Background` one at a time, from `MainWindow` down to `FolderView` — `PaneGroupView.RootGrid` filled correctly at every test; `TabView` was the one link in the chain that didn't. `FolderView.xaml`'s `FileListArea` wrapping `FileList` (`ContextFlyout`/`AllowDrop`/`DragOver`/`Drop` live there, wired via `AddHandler(..., handledEventsToo: true)` in the constructor rather than plain XAML attributes) and the `TabViewItem`/`FolderView` `Stretch` alignments in `PaneGroupView.xaml` all turned out to be unnecessary for this specific bug, but are harmless and left in place as defensive redundancy — don't go looking for a second bug there if something in this area misbehaves again; check `TabView.VerticalAlignment` first.
